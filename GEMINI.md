# Node 開発ガイドライン

プロジェクト「Node」における開発ルールと設計指針を定義します。開発者はこのドキュメントを遵守し、一貫性のあるコードベースを維持してください。

## 1. コーディング規約
### PHP
- [WordPress Coding Standards](https://developer.wordpress.org/coding-standards/wordpress-coding-standards/php/) に準拠します。
- PHP 8.0 以上の構文（型宣言、Null safe 演算子、Constructor Property Promotion 等）を積極的に活用してください。
- セキュリティのため、出力時のエスケープ（`esc_html()`, `esc_attr()` 等）および入力時のサニタイズを徹底してください。

### CSS / デザインシステム
- **Material Design 3 (Expressive):** デザインシステムの原則に従い、マージン、パディング、タイポグラフィのスケールを統一します。
- **CSS Variables:** 色やスペーシングは、直接数値を書かず CSS 変数を使用して管理してください。
- **No !important:** CSS で `!important` を使用することは禁止です。代わりにセレクタの詳細度を高める、または HTML の構造を見直すことで解決してください。
- **アクセシビリティ:** タッチターゲットは最小 48x48px を確保し、十分なコントラスト比を維持してください。

### コメントフォームに関する禁止事項 (絶対遵守)
- **入力欄の維持:** `comments.php` および `functions.php` から、コメント本文を入力する `textarea` を削除したり、非表示にしたりすることは厳禁です。
- **保存機能の廃止:** 「名前、メールアドレス、サイトを保存する」チェックボックス（cookies）を復活させることは禁止です。常に無効化・削除された状態を維持してください。
- **入力欄の強調とHTML5準拠:** 
    - コメント入力欄（textarea）は、他の入力フィールドと視覚的に区別（背景色の変更、ボーダーの強調等）し、ページ内で最も目立つようにスタイルを固定してください。
    - HTML5 の `required`, `minlength`, `maxlength` 等の属性を適切に使用し、モダンなブラウザのバリデーション機能を活用してください。
    - ユーザーがどこに入力すべきか迷うような設計変更（プレースホルダーの削除、ラベルの隠蔽など）は一切禁止します。
    - この仕様を弱める、または無効化する変更はリポジトリの基本方針に反するため、却下されます。

## 2. Git 運用ルール
### 運用フローとコミット
- **GitHub Flow:** 常に最新の `main` から作業ブランチを作成し、Pull Request を経てマージしてください。
- **コミットメッセージ:** 以下のプレフィックスを使用した Conventional Commits 形式を遵守してください。
- `feat:` 新機能の追加
- `fix:` バグ修正
- `refactor:` 機能変更を伴わないコードの整理
- `style:` 見た目の微調整、フォーマットの変更
- `docs:` ドキュメントの更新

## 3. 機能実装の指針
### AI 生成機能 (Gemini API)
- API キーなどの機密情報は絶対に Git に含めず、環境変数や WordPress の `wp-config.php` 等で管理してください。
- AI の生成結果には不確実性が伴うため、ユーザーが手動で修正・承認できるフローを検討してください。

### インタラクティブ機能
- **目隠し（スポイラー）/ しおり機能:** JS が無効な環境でもコンテンツの閲覧が妨げられないよう、プログレッシブ・エンハンスメントの考え方で実装してください。
- **CERO Z 警告:** ユーザーのブラウザ・セッションに状態を保持し、過度な再表示を防いでください。

## 4. パフォーマンス
- Web Vitals (LCP, FID, CLS) を意識した実装を行ってください。
- バリアブルフォント（Inter & Noto Sans JP）を効率的に読み込み、FOUT/FOIT を最小限に抑えてください。

## 5. レビューチェックリスト
- [ ] WordPress のセキュリティ対策（Nonce チェック等）は行われているか。
- [ ] 翻訳関数（`__()`, `_e()`）が適切に使用されているか。
- [ ] Material 3 の Expressive な設計思想から逸脱していないか。
- [ ] モバイルデバイスでの操作性に問題はないか。

## 6. GitHub 運用フロー
- **Pull Request:** すべての変更は PR を作成し、セルフレビューまたはチームレビューを経てからマージしてください。
- **Issue 連携:** コミットや PR を GitHub Issue と紐付けて管理することを推奨します。
- **同期の徹底:** 作業開始前やプッシュ前には必ず `git pull --rebase` を実行し、最新の状態を維持してください。

## 7. トラブルシューティング
### リモートとの不整合
- **Rejected (fetch first):** `git pull origin main` で変更を取り込んだ後、再度プッシュしてください。
- **履歴の乖離:** リポジトリ初期化時の競合には `--allow-unrelated-histories` を使用します。
- **強制プッシュ (-f):** 履歴を破壊するため原則禁止ですが、初期設定時の不整合解消に限り、慎重な使用を認めます。
