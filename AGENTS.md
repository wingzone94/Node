## 命名・ブランドの正本

* ブログ / サイトのブランド名は **Luminous Core**
* WordPress テーマ名は **Node**
* `style.css` の `Theme Name`、テーマディレクトリ名、配布ZIP内のルートディレクトリ名は **Node** に統一すること
* ヘッダー、フッター、OGP、メタ情報、サイト表示、`get_bloginfo( 'name' )` 由来の表示は **Luminous Core** を使用すること
* 「Luminous Core テーマ」「Luminous Core（Node）テーマ」のように、ブログ名とテーマ名を混同しないこと
* 併記が必要な場合は「ブログブランド: Luminous Core / テーマ: Node」と明記すること

* **作業に取りかかる前に、必ずこの AGENTS.md を一読すること**

## AIエージェント運用ルール【重要】

このリポジトリでは、AIエージェントの暴走・過剰なコンテキスト消費・不要な大規模変更・週次制限到達を避けるため、以下のルールを必ず守ること。

---

### 1. 1タスク1目的

* 1回の作業では、原則として **1つの目的だけ** に集中すること
* 「テーマ全体を最適化」「CSS全体を整理」「全部直して」のような曖昧な大規模作業は禁止
* 複数の不具合を扱う場合は、必ずタスクを分割すること
* 調査、修正、レビュー、ZIP生成はできるだけ分けること

悪い例:

```text
Nodeテーマ全体を最適化してください。
```

良い例:

```text
投稿本文コンテナの標準幅と中央配置だけを修正してください。
```

---

### 2. 読み込むファイル数の制限

* 1回の作業で読み込むファイルは、原則として **最大5ファイルまで**
* 6ファイル以上を読む必要がある場合は、先に理由を説明し、ユーザーの確認を待つこと
* リポジトリ全体の横断調査は、ユーザーから明示的に指示された場合のみ行うこと
* 関係のないファイルを開いたり、コンテキストに含めたりしないこと
* 既に調査済みの内容を、毎回最初から調査し直さないこと
* AGENTS.md、過去の調査結果、複数ファイル全文、長いdiffを同時に大量投入しないこと

---

### 3. コンテキスト肥大化の回避

* 会話コンテキストが肥大化している場合は、新しいチャットまたは新しいComposerで作業を分割すること
* 長大なプロンプト、全体調査、多数ファイル読み込みを繰り返さないこと
* コンテキスト使用量が高い場合は、作業を止めてタスク分割を提案すること

目安:

* 50%超: 注意。不要なファイルや過去文脈を削る
* 70%超: 新しいチャットまたはComposerへの分割を検討
* 80%超: 原則として現在の作業をまとめ、新しいセッションに移行する
* 90%超: 追加作業禁止。要約を作成して新規セッションへ移る

## モデル使用方針・週次制限対策【最重要】

このリポジトリでは、AIモデルの週次制限・月次利用枠・コンテキスト消費を守るため、GPT-5.5、GPT-5.3 Codex、Opus 4.8 などの高性能モデルを無制限に使用しないこと。

高性能モデルは「常用モデル」ではなく、設計判断・重要修正・破壊的変更前レビューのための限定リソースとして扱う。

---

### 高コストモデルとして扱うもの

以下のモデルは高コストモデルとして扱う。

* GPT-5.5
* GPT-5.5 Thinking
* GPT-5.3 Codex
* GPT-5.3 Codex High / Thinking 系
* Claude Opus 4.8
* Claude Opus 4.8 thinking-high
* Claude Opus 4.8 thinking-high-fast
* その他、週次制限・月次制限・高トークン消費が発生しやすいモデル

---

### 高コストモデルを使ってよい場面

高コストモデルは、以下の場合に限定して使用すること。

* 難しい設計判断
* WordPressテーマ全体に影響する修正方針の決定
* 破壊的変更前のレビュー
* 複雑なバグの原因調査
* CSS / PHP / JS / theme.json の依存関係が絡む高度な判断
* セキュリティ、データ破損、Fatal error の可能性がある修正
* リリース直前の最終レビュー
* Codex、Cursor、AntiGravity の軽量モデルが詰まった場合の限定的な再確認

---

### 高コストモデルを使わない場面

以下の作業では、高コストモデルを使わないこと。
Auto、軽量モデル、通常モデル、手動確認を優先する。

* typo修正
* CSS数行〜数十行の小規模修正
* クラス名の軽微な調整
* コメント修正
* READMEやリリースノートの下書き
* コミットメッセージ作成
* 単純なdiff要約
* 既に原因が判明している修正
* 既に調査済みの内容の再確認
* 同じAGENTS.mdの再読
* 同じファイル群の再調査
* 記事執筆、文章整理、アイデア出し

---

### GPT-5.5 / GPT-5.3 Codex 使用制限

GPT-5.5 および GPT-5.3 Codex は、週次制限に到達しないように以下のルールで使用すること。

* 1回の依頼では、原則として1目的のみ扱う
* 1回の依頼で読み込むファイルは原則5ファイルまで
* 6ファイル以上を読む必要がある場合は、先に理由を説明し、ユーザーの確認を待つ
* リポジトリ全体の横断調査は禁止
* 「テーマ全体を最適化」「全部直して」「不要コードを全部整理」などの広すぎる依頼は禁止
* 1回の依頼でCSS、PHP、JS、theme.jsonを同時に大きく変更しない
* 長大なプロンプトを毎回使い回さない
* AGENTS.md、過去の調査結果、複数ファイル全文、長いdiffを同時に大量投入しない
* 既に調査済みの内容を毎回最初から調査し直さない
* 変更後は必ずdiffを確認し、追加修正を繰り返す場合はタスクを分割する

---

### Codex向け運用ルール

Codexを使用する場合は、特に以下を守ること。

* 1回のCodexタスクは小さくする
* 1タスクにつき1つの修正だけを依頼する
* 「調査だけ」「修正だけ」「レビューだけ」を分ける
* 大規模リファクタリングと不具合修正を同時に依頼しない
* 週次制限が近い場合は、Codexをレビュー専用または重要修正専用に切り替える
* CodexにZIP生成まで自動で進ませる場合は、ビルド・cybernode.local検証が成功した場合のみ許可する
* Codexが不要に大きな差分を出した場合は、採用せずタスクを分割してやり直す
* 制限回避のために、同じ巨大タスクを別モデルへ横流ししない

---

### Cursor / Opus向け運用ルール

CursorまたはOpus系モデルを使用する場合は、以下を守ること。

* Opus 4.8 thinking-high / thinking-high-fast は常用しない
* 小規模CSS修正やdiff確認では Auto または軽量モデルを使用する
* 1回のComposerで大量のファイルを開き続けない
* Context Usage が70%を超えた場合は、新しいComposerへの分割を検討する
* Context Usage が80%を超えた場合は、原則として作業をまとめ、新しいComposerへ移る
* 14ファイル以上の同時読み込みや、160K tokens級のコンテキスト消費を繰り返さない
* AGENTS.mdを毎回全文再読させない
* 調査済み内容は短く要約して次タスクに渡す

---

### AntiGravity向け運用ルール

AntiGravityを使用する場合は、以下を守ること。

* Local Modeで本体を直接大改修しない
* 可能な限りWorktree Modeを使う
* 最初は調査のみを依頼する
* PHPテンプレート、functions.php、Vite設定を同時に大きく触らせない
* 変更前に差分・計画を確認する
* 本番相当ファイルへ反映する前に必ずcybernode.localで確認する

---

### 高コストモデルの使用上限目安

明確な許可がない限り、1日の高コストモデル使用は以下を目安に抑えること。

* 大きな調査: 1日1回まで
* 重要修正: 1日1〜2件まで
* リリース前レビュー: 必要時のみ
* 軽微な修正: 高コストモデル使用禁止

週次制限に近い場合、または使用量が急増している場合は、以下の運用に切り替える。

* 高コストモデルの新規使用を停止
* Auto / 軽量モデル中心にする
* ChatGPT側でプロンプトや仕様を整理してから投げる
* Codexは最重要タスクだけに使う
* 作業を翌週または制限解除後に回す

---

### 高コストモデル使用前チェック

GPT-5.5、GPT-5.3 Codex、Opus 4.8 などを使う前に、以下を確認すること。

* この作業は本当に高コストモデルが必要か
* Autoや軽量モデルで十分ではないか
* 対象ファイルは5ファイル以内か
* 目的は1つに絞れているか
* 既に調査済みの内容を再調査していないか
* 変更範囲が広すぎないか
* cybernode.localで検証できる範囲か
* 週次制限・月次利用枠を浪費しない内容か

上記のいずれかに問題がある場合は、高コストモデルを使わず、タスクを分割すること。

## 開発ルーチン（ビルド → デプロイ → 検証 → ZIP）

すべてのテーマ変更は、以下の順序で行うこと。
**ZIP出力はテスト確認後にのみ実行すること。**

---

### Step 1: コード変更・ビルド

```bash
bun x vite build
```

注意:

* ビルドには必ず `bun` を使うこと
* `npm`、`yarn`、`pnpm` は使用禁止
* `vite.config.js` が存在しない場合は新規作成しないこと
* ビルドエラーが出た場合は、ZIP生成やデプロイに進まないこと

---

### Step 2: ローカルテスト環境（cybernode.local）へデプロイ

```bash
# テーマファイルを同期（ZIPではなく直接コピー）
rsync -a \
  --exclude='.git/' \
  --exclude='node_modules/' \
  --exclude='*.zip' \
  --exclude='.DS_Store' \
  --exclude='.cursor/' \
  --exclude='.gemini/' \
  --exclude='scratch/' \
  --delete \
  ./ "/Users/saitoutatsuya/Local Sites/cybernode/app/public/wp-content/themes/node/"

# 【必須】plugins-embedded を削除（プラグインとの二重読み込みを回避）
rm -rf "/Users/saitoutatsuya/Local Sites/cybernode/app/public/wp-content/themes/node/plugins-embedded"

# OPcache 対策: 変更ファイルの mtime を更新
find "/Users/saitoutatsuya/Local Sites/cybernode/app/public/wp-content/themes/node/" \
  -name '*.php' -newer /tmp/.node_last_deploy -exec touch {} + 2>/dev/null || true
touch /tmp/.node_last_deploy
```

---

### Step 3: cybernode.local で動作確認

ブラウザまたはPuppeteerで、以下を確認すること。

* `http://cybernode.local`
* 通常投稿ページ
* 固定ページ
* アーカイブページ
* 検索結果ページ
* 画像・埋め込み・テーブルを含む投稿
* 目次・ブログカード・関連記事を含む投稿
* スマートフォン幅の表示

Fatal error 等がないことを以下で確認すること。

```bash
curl -s http://cybernode.local/ | grep -i 'error'
```

注意:

* 問題があれば Step 1 に戻って修正すること
* 問題が残っている場合、ZIP出力に進まないこと
* `cybernode.local` で確認できなかった場合は、検証済みと書かないこと
* 検証できなかった理由を報告すること

---

### Step 4: テスト確認後に ZIP 出力

テスト環境で問題がないことを確認してから、`HOW_TO_RELEASE.md` に従いZIPを生成する。

```bash
rm -f node.zip
repo_dir=$(pwd)
tmpdir=$(mktemp -d)
rsync -a \
  --exclude='.git/' \
  --exclude='node_modules/' \
  --exclude='*.zip' \
  --exclude='.DS_Store' \
  --exclude='.!*!.DS_Store' \
  --exclude='.cursor/' \
  --exclude='.gemini/' \
  --exclude='scratch/' \
  ./ "$tmpdir/node/"
(cd "$tmpdir" && zip -qr "$repo_dir/node.zip" node)
rm -rf "$tmpdir"
```

---

## テスト環境に関する注意事項

* **テスト環境パス**: `/Users/saitoutatsuya/Local Sites/cybernode/app/public/wp-content/themes/node/`
* **テスト環境URL**: `http://cybernode.local`
* **plugins-embedded の競合**:

  * テスト環境では `node-signal`・`node-flow`・`node-ai-tools` がシンボリックリンクで `/wp-content/plugins/` にも存在する
  * テーマ側の `plugins-embedded/` を残すと `Cannot redeclare` Fatal Error が発生する
  * デプロイ後に必ず `plugins-embedded/` を削除すること
* **OPcache**:

  * LocalWP の PHP が古い `.php` をキャッシュする場合がある
  * 変更が反映されない場合は `touch` でタイムスタンプを更新するか、LocalWP のPHPを再起動すること

## 軽量化・刷新の基本方針

---

### 1. まず棚卸し

必要な場合のみ、以下を棚卸しする。

* `src/main.js` 内の機能分類

  * ページ共通
  * 単一記事
  * ホーム / アーカイブ
  * 検索
  * TOC / FAB
  * 管理・補助機能
* PHP側の出力とJS/CSS側のセレクタ照合
* 実際に使われていないUIとコードの特定
* 古いハッシュ付き成果物、古いZIP、作業用ディレクトリ、`.DS_Store` の削除候補確認

ただし、棚卸しは高コストになりやすいため、ユーザーから明示的に依頼された場合、または対象タスクに必要な場合のみ行うこと。

---

### 2. `src/main.js` のモジュール化

* 既存の `src/scripts/` を活用し、機能単位で切り出す
* 推奨分割例:

  * `search-bar`
  * `smart-header`
  * `expressive-toc`
  * `article-navigation`
  * `home-layout`
  * `scroll-animations`
  * `comments`
  * `header-clock`
* `src/main.js` は import と初期化順序の管理に寄せる
* 既存挙動を残す必要がある場合も、重複する旧ロジックと新ロジックが同時に走らないようにする

注意:

* モジュール化は有効だが、一度に大規模に行わないこと
* 1回の作業では、1機能単位で分割・修正すること

---

### 3. 条件付き初期化・条件付きロード

* 単一記事だけで必要な処理は `body.single` / `body.single-post` などで限定する
* ホーム・アーカイブ専用処理は該当ページでのみ動かす
* 検索モーダル、TOC、読了進捗、記事ページ用UIなどはDOMの存在確認だけでなく、ページ種別も見て初期化する
* 可能であれば dynamic import を使い、初回ロードで不要な機能を読み込まない

---

### 4. CSS整理

* `src/styles/style.css` の import 構成を正本とする
* 未使用セレクタは PHP / JS 出力と照合してから削除する
* 追加パッチを末尾に積み続けず、責務に合うpartialへ移す
* `!important` は既存仕様維持に必要な場合に限定し、刷新時には段階的に減らす
* ただし、CSS整理は広範囲に波及しやすいため、1回の作業では対象を限定すること

---

### 5. 残骸整理

* 古い配布ZIP、作業用ディレクトリ、不要なハッシュ付き成果物、`.DS_Store`、一時スクリーンショットは削除候補にする
* 削除前に、`functions.php`、`inc/`、`template-parts/`、Vite manifest、ライブHTMLとの参照関係を確認する
* 配布ZIPに含めるべきものと、リポジトリ作業用に留めるものを分ける
* 残骸整理は、機能修正タスクとは分けて実施すること

## WordPress ブロック / レイアウト修正ルール

投稿本文・固定ページ・ブロックエディタ由来の表示を修正する場合は、以下を守ること。

以下のWordPress標準クラスは尊重すること。

* `.has-text-align-center`
* `.has-text-align-left`
* `.has-text-align-right`
* `.aligncenter`
* `.alignleft`
* `.alignright`
* `.alignwide`
* `.alignfull`

---

### 本文レイアウトの基本

* 既存記事の表示を壊さないこと
* 本文コンテナは自然に中央配置する
* 本文テキストは原則左揃えにする
* 画像、figure、embed、ブログカード、目次、関連記事は本文幅内で自然に収める
* 「全部を中央揃え」にしない
* `text-align: center;` を本文全体へ安易に適用しない
* スマートフォンで横スクロールを発生させない
* CSSだけで解決できる場合は、PHPテンプレートを変更しない
* JavaScriptでレイアウトを後付け制御しない

---

## 検証方針

変更後は、最低限以下を確認する。

1. `bun x vite build`
2. `assets/.vite/manifest.json`、`assets/js/*`、`assets/css/*` が更新内容と整合していること
3. cybernode.local へ同期
4. トップページ確認
5. 単一記事確認
6. 検索ページ確認
7. TOC / FAB確認
8. ホーム / アーカイブ表示確認
9. スマートフォン幅確認
10. `curl -s http://cybernode.local/ | grep -i 'error'` で Fatal error 等が出ていないこと

検証できなかった項目は、必ず「未検証」として理由を書くこと。

## 禁止事項まとめ

以下は禁止する。

* AGENTS.mdを読まずに作業開始すること
* 1回の作業で目的を複数詰め込むこと
* 原則5ファイルを超える無断読み込み
* リポジトリ全体の無断調査
* GPT-5.5 / GPT-5.3 Codex / Opus 4.8 で長大な全体調査を繰り返すこと
* 高コストモデルでリポジトリ全体を無断調査すること
* 週次制限が近い状態で、大規模リファクタリングや全体最適化を依頼すること
* 軽微なCSS修正やdiff要約に高コストモデルを使うこと
* 制限回避のために別モデルへ同じ巨大タスクを横流しすること
* ファイル全体の自動整形
* 関係ないファイル変更
* 未使用CSS / JS の一括削除
* PHPテンプレート構造の大幅変更
* TypeScript化
* フレームワーク導入
* `!important` の乱用
* CSS、PHP、JS、theme.json を1回のタスクでまとめて大きく変更すること
* Codexに「全部直して」「全体を最適化して」のような曖昧な依頼を投げること
* 検証していない内容を検証済みと報告すること
* cybernode.localで確認できなかった内容を確認済み扱いにすること
* テスト前にZIPを出力すること

## 最終原則

このリポジトリにおけるAIエージェントの役割は、Nodeテーマを安全に改善することであり、無制限にリポジトリ全体を探索・改変することではない。

常に以下を優先すること。

1. 既存記事を壊さない
2. 変更範囲を狭くする
3. 高コストモデルを浪費しない
4. cybernode.localで確認する
5. 検証していないことを検証済みにしない
6. ZIPはテスト後にのみ生成する
7. Luminous Core と Node のブランド命名を混同しない
