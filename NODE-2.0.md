## 0. テーゼ（2.0 の人格）

**「コンテンツの色をまとうブログ」。**
記事のアイキャッチから生成した色で**コンテンツ領域（surface / accent / リンク / カード / バッジ）**が着替える。
ただし**ヘッダーとフッターは既存のブランドオレンジで固定**し、色をまとう中身を引き締める“額縁”とする。
オレンジは “Luminous Core のブランド既定色” として、ヘッダー/フッターおよび**画像由来の色が無い場面のフォールバック**に位置づける。
これにより「オレンジを差し色にした Material 3 テーマ」から「記事ごとに表情が変わる唯一無二のテーマ」へ転換する。

## 1. 現状の到達点と差分（なぜ今“効いていない”か）

| 項目 | 現状 | 2.0 で必要 |
| --- | --- | --- |
| 抽出 | ColorThief で単一ドミナントRGB（`src/colorExtractor.js`） | seed色 → **M3トーナルパレット生成**（primary/secondary/surface/on-* を導出） |
| 適用範囲 | カテゴリ/読了バッジのみ（`src/scripts/color-extraction.js`） | **ページ単位**（`:root`/`body` のCSS変数を上書き） |
| タイミング | クライアント実行のみ・保存なし | **保存時にPHPで先行抽出 → 投稿メタ保存 → `<head>` にインライン出力**（FOUC/CLS回避）。JSは未保存記事のフォールバックに降格 |
| 配色思想 | 全面オレンジ | オレンジは**フォールバック専用**。動的色を主役に |
| ダーク対応 | バッジ固定 | **スキーム別にトーナルマッピング**（ライト/ダークで別トーンを選ぶ） |
| a11y | 文字白固定＋影で誤魔化し | **on-color をコントラスト保証付きで導出**（トーナルパレットで担保） |

## 2. カラーシステム仕様

### 2.1 役割トークン（`src/styles/_variables.css` に追加）
seed色から以下を導出し、CSS変数で公開する。既存の固定オレンジ変数は**フォールバック値**として残す。

```
--lc-seed            /* 抽出された種色 */
--lc-primary / --lc-on-primary
--lc-secondary / --lc-on-secondary
--lc-surface / --lc-surface-variant / --lc-on-surface
--lc-accent          /* リンク・CTA・現在地に限定使用 */
--lc-outline
```

### 2.2 適用先（ページ単位）

**大原則: ヘッダーとフッターは動的カラーの対象外。既存のブランドオレンジで固定する。**
動的色が効くのは**ヘッダーとフッターに挟まれたコンテンツ領域だけ**。固定の額縁（chrome）が、色をまとう中身を引き締める。

- **ヘッダー / フッター（固定）**: 既存のプライマリ＝ブランドオレンジのまま。seed色・役割トークンを**一切流し込まない**。`--lc-*` をここに適用しない。
- 単一記事（動的）: ヒーロー／本文リンク／TOCアクティブ／FAB／関連記事アクセント／カテゴリバッジ
- ホーム・アーカイブ（動的）: 各カードが**その記事の seed色**をアクセントに持つ → グリッドが「単色オレンジ」から「スペクトラム」になる（=2.0の見せ場）

### 2.3 抽出パイプライン（FOUC/CLS を出さない）
1. **保存時（PHP）**: アイキャッチ確定時に seed色を算出し `post_meta`（例 `_lc_seed_color`）へ保存。
2. **表示時（PHP）**: 投稿の seed色から役割トークンを計算し、`<head>` に `<style>:root{--lc-…}</style>` をインライン出力 → **初回描画から正しい色**。
3. **フォールバック（JS）**: メタ未保存の旧記事のみ、既存 `colorExtractor.js` 経路でクライアント抽出。
   - 既存のバッジ専用ロジックは、ページ単位適用へ拡張 or 置換する。

### 2.4 トーナル変換
- 生RGB直貼りをやめ、**HCT または HSL ベースでトーン階調を生成**（M3準拠）。
- ライト: 高明度surface＋中明度primary。ダーク: 低明度surface＋やや高彩度primary。
- on-color は背景トーンに対し WCAG AA を満たすトーンを選択（白固定をやめる）。

## 3. 段階的ロールアウト（Codex 用・1タスク1目的）

> **前提:**
> - v1.1.1（脚注/ページネーション/SEO）が完了・リリースされてから 2.0 着手。

- **Phase 0（コストゼロ・先行可 / 親テーマ node 側）**: `node.zip` を git 管理から除外、デッドCSS（`_404.css`/`.ai-summary-card.css`）の配線確認と整理、`TECHNOLOGIES.md` のバージョン更新。
- **Phase 1（子テーマ scaffold）**: `node-next` 子テーマの骨組みを作成（§6.2）。親を継承するだけで親と同一表示になることを cybernode.local で確認（=継承が正しく効く土台確認）。
- **Phase 2（トークン土台）**: 役割トークンを子の `_variables.css` で定義（固定オレンジはフォールバックへ）。**この時点では見た目を変えない**。
- **Phase 3（seed保存・適用）**: PHP で seed色の保存（save_post フック）＋ `<head>` インライン出力。単一記事のみ適用。旧記事・新記事・画像なし記事を検証。
- **Phase 4（JS フォールバック拡張）**: JS フォールバックを「バッジ専用」から「ページ単位」へ拡張し、旧記事互換を確認。
- **Phase 5（スペクトラム化）**: ホーム/アーカイブのカードを seed色アクセント化。
- **Phase 6（仕上げ）**: ダークモードのトーナルマッピング、a11y コントラスト実測、CSS条件分割（記事専用CSSの分離）。

各Phaseは独立でビルド→cybernode.local検証→問題なければ次へ。

## 4. ガードレール（“ありきたりで賑やか”への退行を防ぐ）

- **ヘッダー / フッターは固定オレンジ。動的色を絶対に流し込まない**（額縁として全体を引き締める。ユーザー明示指示。覆さないこと）。
- **カテゴリラベルの固有色は道標（wayfinding）として残す**（ユーザー明示指示）。これは動的カラー（画像由来）とは別レイヤーの「機能的な色」。既存実装どおり、設定済みの固有色（`data-color` の hex）が画像抽出色に優先する（`src/scripts/color-extraction.js`）。動的色とカテゴリ固有色が衝突しないよう、動的色は surface/accent に効かせ、カテゴリラベルは固有色を保つ。
- カードの**二次的なタグチップ（カテゴリラベル以外）は数を絞る／彩度を落とす**。カテゴリ固有色を主役にし、周辺のタグは静かにしてコントラストを作る（多色が張り合うと肝心のカテゴリ色が埋もれる）。
- 動的色は **surface と accent の役割にだけ**効かせる。全要素を派手に塗らない（現状の失敗の逆をやらない）。
- アクセント色は **CTA・リンク・現在地**に限定。本文テキストは中立色。
- `!important` を増やさない。既存記事のレイアウトを壊さない。
- エフェクト追加で個性を出そうとしない。**個性は「色が内容に従う」一点に集約**する。

## 6. 並行開発とテスト分離設計（v1.1.2 ∥ 2.0試作）

> **方針: 2つの開発線を物理的に別テーマへ分離し、テスト環境で同時に並べてテストできるようにする。実装はまだ行わない（本章は設計のみ）。**

### 6.1 2系統の役割

| 線 | テーマ | 担当 | 出荷 |
| --- | --- | --- | --- |
| 安定線 | **親テーマ `node`**（既存） | v1.1.2（Codex 主力） | 本番出荷線 |
| 実験線 | **子テーマ `node-next`**（新規・Template: node） | 2.0 試作（§3 Phase 1〜6） | 検証通過まで本番統合しない |

**継承の核心（ユーザー要望そのもの）:** 子テーマは親 `node` を継承するため、**v1.1.2 で親に入る修正が `node-next` へ自動で流れ込む。** 独立フォークのようなドリフト・二重メンテが発生しない。

**副次効果:** §4 の「ヘッダー/フッターは固定オレンジ」は、子が親の `header.php` / `footer.php` をそのまま継承することで**構造的に自動保証**される（子はこれらのテンプレを持たない＝親chromeをそのまま使う）。動的色は子が上書きするコンテンツ領域にしか及ばない。

### 6.2 子テーマ `node-next` の最小構成（実装時の指針）

- `style.css`: `Template: node` / `Theme Name: Node Next (2.0 Prototype)` / `Version: 2.0.0-alpha`
- `functions.php`: 親スタイル → 子の 2.0 デルタCSS の順で enqueue。2.0 の動的カラー PHP（save_post での seed 抽出、`<head>` インライン出力）はここに追加。
- 2.0 デルタのみを持つ:
  - `_variables.css` の役割トークン上書き＋コンテンツスコープCSS
  - ページ単位適用版の動的カラー JS
- **`header.php` / `footer.php` / 大半のテンプレは持たない**（親を継承＝固定オレンジ維持）。2.0 で構造変更が必要なテンプレだけを子へコピーして上書き。
- ビルド: 子テーマ用に **Vite のエントリ/出力を分離**し、親 `assets/` を壊さない。子は親アセットを読みつつ 2.0 デルタ資産だけ追加ビルドする。

### 6.3 テスト環境（cybernode.local）への配置

- 親を `themes/node/`、子を `themes/node-next/` の**2ディレクトリへ同期**（AGENTS.md の rsync 手順を2系統に拡張）。
- WP 管理画面に2テーマが並ぶ → **有効化を切り替えるだけでどちらも単独テスト可**。別ディレクトリなので並行編集衝突なし。
- `plugins-embedded/` 削除ルールは従来どおり（親側）。子 `node-next` は `plugins-embedded/` を持たない。

### 6.4 昇格規律

- `node-next` はあくまで試作。§5 の検証が全Phaseで通るまで本番テーマ `node` へ統合しない。
- 2.0 として出荷する段階で初めて、子の成果を親へマージするか、子を新本番テーマへ昇格させるかを判断（その判断は別途 Opus 前段で行う）。
